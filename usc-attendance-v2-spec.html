01Changelog v1 → v2.0

NEW

QR-Based Check-In Major

Staff can scan officer QR codes instead of manual entry. Each QR contains a signed, time-limited token (60s expiry) validated server-side via HMAC-SHA256.

  • /qr page — officer generates a QR token for a selected event
  • /scan page — staff scans QR using device camera (html5-qrcode)
  • /attendance/qr API — validates token signature + expiry before recording
  • QR secret stored in VITE_QR_SECRET env var
NEW

Identity Lifecycle Major

Complete officer onboarding and lifecycle management. No public registration — all accounts are provisioned by the secretary/super_admin from the official roster.

  • mustChangePassword field on User — forces password change on first login
  • active field on User — soft-delete for officer graduation/transition
  • CSV Import — bulk officer provisioning from official roster
  • Supabase Auth password recovery flow (email reset link)
  • Yearly transition workflow: deactivate outgoing → import incoming → go live
NEW

Audit Log New Table

Lightweight audit table tracking who changed what and when.

  • Logs: account created, password reset, officer deactivated, role changed, manual attendance override
  • performedBy captures the actor (admin/system)
NEW

Office Duty Scheduling Major

Recurring daily duty tracking for officers assigned to regular office hours (Mon–Sun). Officers check in/out for daily duty separate from event attendance.

  • OfficeDuty model — links user to day-of-week + time range
  • /duty page — officer views today's duty and checks in
  • /attendance/duty API — records duty attendance with date-based uniqueness
  • Attendance type split: type: "event" vs type: "duty"

02Scope of v2.0

v2.0 adds QR check-in, office duty scheduling, and full identity lifecycle management. Accounts are provisioned by the secretary/super_admin from the official roster — no public registration. Officers authenticate, change password on first login, and use authenticated, short-lived QR tokens for check-in. The system supports the complete yearly officer transition cycle.

In scope

  • 3 roles: super_admin, staff (officer-on-duty), client (officer)
  • Officer provisioning via CSV import (no public registration)
  • Force password change on first login (mustChangePassword)
  • Password recovery via Supabase Auth email reset
  • Account soft-delete via active flag (preserves attendance history)
  • Audit log for account and attendance changes
  • Event creation with location + radius
  • Self check-in via geolocation match (event)
  • QR check-in via signed token scan (event)
  • Manual override check-in by staff with recorded_by audit
  • Office duty scheduling per day-of-week + time range
  • Self duty check-in for scheduled officers
  • Offline queueing of check-ins (event and duty), synced on reconnect
  • Attendance list/export per event (CSV)
  • PWA install + offline app shell
  • Basic auth (email/password or magic link via Supabase Auth)
  • HMAC-signed QR tokens with 60s expiry

Explicitly not in v2.0

  • Push notifications / reminders
  • Analytics dashboards, charts, streak tracking
  • Multi-org / multi-council support
  • Editing or deleting attendance records (append-only)
  • Background location tracking
  • Native app wrapper (Capacitor) — PWA only
  • CI/CD pipelines, automated test suites
  • Duty schedule management UI (staff create/edit duties) — seeded via Prisma seed only
  • Duty export or reporting beyond per-day view
  • Face recognition, NFC — geolocation and QR only
  • Duty geolocation enforcement — location not required for duty check-in
Rule of thumb

If a feature isn't required to answer "was this officer present at this event or duty, yes or no, recorded by whom, and how?" — it doesn't belong in v2.0.

Architecture note

Prisma bypasses Supabase RLS. Don't run both RLS policies and Prisma role checks — pick Prisma + code-level checks in API routes as the single enforcement point, and leave RLS off (or default-deny) for tables Prisma touches.

QR Token Security

QR tokens are signed with HMAC-SHA256 using a secret stored in VITE_QR_SECRET. The secret must be at least 32 characters, randomly generated, and never committed to git. Token expiry (60s) is enforced server-side — do not trust client-side countdown alone.

05Identity Lifecycle — Account Provisioning

This is the most important operational workflow in the system. Every officer is already known — there is no public registration. Accounts are created by the secretary or super_admin from the official roster.

Officer Provisioning Flow

How an officer becomes a user

Secretary compiles official roster → Super Admin imports via CSV → System creates accounts → Temporary password generated → Credentials sent privately → Officer logs in → Forced password change → Account activated

User Model — New Fields

FieldTypeDefaultPurpose
mustChangePasswordBooleantrueForces password change on first login. Set to false after successful change.
activeBooleantrueSoft-delete flag. Set to false at end of academic year to preserve attendance history while blocking login.

Account Provisioning Rules

  • No public registration — login page is the only entry point
  • Accounts are created via CSV import or prisma db seed — not by officers themselves
  • Initial password is randomly generated (e.g. USC2026!Xr2) and shared privately by the secretary
  • mustChangePassword: true on account creation — officer must change password before accessing any other page
  • If active: false, login is blocked and officer sees "Your account has been deactivated" message
  • Password recovery uses Supabase Auth's built-in email reset flow — no admin intervention needed
  • All account operations (create, deactivate, role change, password reset) are logged to audit_logs

Role must be one of: super_admin, staff, client. The import generates a temporary password for each officer and writes it to a downloadable CSV for the secretary to distribute.

Audit Log Events

ActionLogged When

Why no permanent QR IDs

Permanent QR codes introduce problems: screenshot sharing, revocation, regeneration, blacklisting. Temporary, signed tokens (60s expiry) eliminate all of these. The officer's authenticated account IS their identity — no printed card needed.

08Anti Over-Engineering Reminders

Before adding anything

Ask: does this help answer "was this officer present, yes or no, recorded by whom, and how did they check in (self/QR/manual)?" If not, it's v2.1 or later.

  • No CI/CD pipeline. Deploy from your machine via Vercel CLI or git push, review it yourself.
  • No automated test suite for v2.0. Manual smoke test per phase is enough at this scale.
  • No edit/delete on attendance rows. Append-only, corrections are new rows.
  • No analytics, streaks, leaderboards, or gamification — these always sound small and never are.
  • No multi-council/multi-org abstraction. Build for ISUFST Dingle Campus. Generalize later if it's ever actually needed.
  • No native app. PWA covers "add to home screen" and offline — that's the whole ask.
  • Resist RLS + Prisma both. Pick one enforcement layer (Prisma + API route checks) and stop there.
  • No duty schedule management UI. Office duties are seeded via prisma/seed.ts only. If staff need to add/edit duties, that's a future feature.
  • No geolocation enforcement for duty check-in. Location is not required for duty attendance — only for event self-check-in.
  • No permanent QR codes or printed ID cards. Temporary, signed tokens (60s) are the identity — the officer's authenticated account is their ID.
  • No self-registration. All accounts are provisioned by the secretary/super_admin from the official roster via CSV import.
  • QR tokens are signed but not encrypted. Don't put sensitive data in the token payload beyond what's already in the DB.
  • If an execution agent proposes a "flexible" or "extensible" abstraction you didn't ask for — reject it. Flexibility you don't need yet is a bug waiting to happen.

09Manual Setup Steps (you, not an agent)

1

Gather ISUFST branding assets

Logos are already in /public/logo/ — ISUFST_logo.png and USC_logo.jpg. If you need alternate versions, get transparent PNGs from the council's official channels.

PWA manifest icons should use the ISUFST logo at 512×512 and 192×192. Update manifest.json if the icon paths change.
2

Generate VITE_QR_SECRET

Generate a cryptographically random string at least 32 characters long. Use a password manager or run:

node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

Add this to Vercel as VITE_QR_SECRET (client-visible, but not secret enough to harm if exposed — the real protection is the HMAC signature).

3

Create the Supabase project

Sign up/log in at supabase.com, create a new project, choose a region close to the Philippines (Singapore is usually closest).

Save: project URL, anon key, service role key, and the session-pooler connection string (port 5432) for Prisma's DATABASE_URL. Also save DIRECT_URL for Prisma's directUrl.
4

Set up the Vercel project

Import the repo into Vercel, set framework preset to Vite, add environment variables.

DATABASE_URL, DIRECT_URL, SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY, VITE_QR_SECRET. Mark service role key as a server-only env var.
5

Run initial Prisma migration

After schema is finalized, run prisma migrate dev --name init (or v2 migration name). This creates the tables including the new office_duties table and updated attendance schema.

Never use db push for production — use migrations only so the schema history is tracked.
6

Get real venue coordinates

For each recurring venue (session hall, gym, quadrangle, office), grab the lat/lng from Google Maps (right-click → coordinates) ahead of time so staff aren't guessing radius values on event-creation day.

Suggested starting radius: 50–75 meters for open areas like a quadrangle, 25–30 meters for smaller rooms.
7

Seed real duty schedules

Edit prisma/seed.ts with the actual officer roster and their weekly duty schedules (dayOfWeek 0–6, startTime, endTime). Run prisma db seed to populate office_duties.

This is the only way to create duties in v2.0 — there is no admin UI for duty management.
8

Compile and import officer roster

Compile the actual officer roster (name, email, position, role) before go-live. Use the CSV import feature in the super_admin dashboard to bulk-create accounts. Each officer gets a temporary password generated by the system.

Example CSV format: name,email,position,role — one officer per line. The secretary distributes credentials privately (email, DM, or printed handout).
9

Test QR scanning on real devices

Browser-based QR scanning behaves differently across platforms. Test the scan flow on at least one Android (Chrome) and one iPhone (Safari) before the first real event. Facebook's in-app browser often blocks camera access — advise officers to open the PWA in the system browser.

html5-qrcode requires HTTPS in production. Localhost works with dev server. The PWA's service worker also requires HTTPS.
10

Test geolocation on real devices

Same as QR — test event check-in via geolocation on real devices. iOS Safari requires the site to be served over HTTPS and needs the user's location permission granted.

High-accuracy mode (GPS) may take longer on some devices. Consider whether to request high-accuracy or accept coarse location for v2.0.
11

Domain (optional)

If ISUFST has or can get a subdomain (e.g. attendance.isufst.edu.ph), point it at the Vercel deployment for credibility instead of a random .vercel.app URL.

10Yearly Transition — Academic Year Rollover

Every academic year, USC officers change. The system must handle this transition cleanly while preserving attendance history.

Annual Officer Transition Process

1

Export attendance archive

Before any changes, export all attendance data as CSV for the outgoing academic year. Store this in the council's records folder as a permanent backup.

The export includes all event and duty attendance with timestamps, methods, and officer details. This is your audit trail for the previous year.
2

Deactivate outgoing officers

Set active: false on all outgoing officer accounts. This preserves their attendance history while preventing future logins. Do NOT delete users — the attendance rows reference their user IDs.

Super admin can bulk-deactivate via the officer management page, or run a script: UPDATE users SET active = false WHERE role != 'super_admin'.
3

Import new officer roster

Receive the new academic year's official roster from the incoming secretary. Use the CSV import feature to create accounts for all new officers.

If an officer is continuing for another term, they need a new account (or their existing account can be reactivated). Do not reuse deactivated accounts — create fresh ones to keep the audit trail clean.
4

Generate and distribute temporary passwords

After import, download the generated credentials CSV. Distribute credentials privately to each new officer — email, DM, or printed handout from the secretary.

Each officer will be forced to change their password on first login (mustChangePassword: true).
5

Update duty schedules

Edit prisma/seed.ts with the new officer IDs and their weekly duty schedules. Run prisma db seed to update office_duties.

Old duty schedules from the previous year should be deactivated (active: false) or deleted. The seed script should handle this cleanly.
6

Go live

New officers log in, change passwords, and start using the system. The first event of the new academic year is the real test.

Monitor the first week closely. If an officer can't log in, check: (1) account exists, (2) active = true, (3) correct email, (4) password reset if needed.
Why soft-delete, not hard-delete

Attendance rows reference user IDs. If you delete a user, their attendance records become orphaned and break referential integrity. Setting active: false preserves history while blocking access. This is standard practice for systems with audit requirements.
