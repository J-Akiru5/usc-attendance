USC Attendance System

Geolocation-gated attendance tracking for University Student Council officers. Vue 3 SPA, PWA-enabled, offline-first, built to ship fast — not to impress anyone with architecture.

01Scope of v1

One job: let an officer prove they physically showed up to an event, and let staff override that when the scanner can't be trusted or used. Everything else is a future version.

In scope

  • 3 roles: super_admin, staff (officer-on-duty), client (officer)
  • Event creation with location + radius
  • Self check-in via geolocation match
  • Manual override check-in by staff, with recorded_by audit field
  • Offline queueing of check-ins, synced on reconnect
  • Attendance list/export per event (CSV)
  • PWA install + offline app shell
  • Basic auth (email/password or magic link via Supabase Auth)

Explicitly not in v1

  • Push notifications / reminders
  • Analytics dashboards, charts, streak tracking
  • Multi-org / multi-council support
  • Editing or deleting attendance records (append-only)
  • Background location tracking
  • Native app wrapper (Capacitor) — PWA only
  • CI/CD pipelines, automated test suites
  • Face recognition, QR codes, NFC — geolocation only
Rule of thumb

If a feature isn't required to answer "was this officer present at this event, yes or no, and who recorded it" — it doesn't belong in v1.

Architecture note

Prisma bypasses Supabase RLS. Don't run both RLS policies and Prisma role checks — pick Prisma + code-level checks in API routes as the single enforcement point, and leave RLS off (or default-deny) for tables Prisma touches.

03Data Model — essentials only

TableKey fields
usersid, name, position, role enum
eventsid, title, date, lat, lng, radius_meters, created_by
attendanceid, event_id, user_id, method self|manual, lat, lng, recorded_by, created_at

Attendance is append-only. No update/delete paths in v1 — if a mistake happens, it gets corrected by a new row, not an edit, so the audit trail stays honest.

04Dev Checklist

Phase 0 — Foundation
  • Supabase project created, connection string saved (session pooler, port 5432)
  • Prisma initialized, schema.prisma written for the 3 tables above
  • First migration run via prisma migrate dev — never db push
  • Vite + Vue 3 + TS scaffold, Tailwind + shadcn-vue installed
  • Vercel project linked, env vars set (DB URL, Supabase keys)
Phase 1 — Auth & roles
  • Supabase Auth wired (email/password or magic link)
  • Role field enforced server-side in every API route (not just UI hiding)
  • Route guards for staff-only and super_admin-only pages
Phase 2 — Core attendance flow
  • Event CRUD (staff/super_admin only)
  • Client check-in screen: geolocation request → Haversine distance calc → allow/deny
  • Server-side distance re-validation on insert (don't trust client-computed distance)
  • Staff manual override screen with recorded_by captured automatically
  • Attendance list per event + CSV export
Phase 3 — PWA & offline
  • vite-plugin-pwa configured, manifest.json with USC branding
  • App shell precached, installable on mobile home screen
  • IndexedDB queue: check-in writes locally first with synced:false
  • Background sync / reconnect listener flushes queue to API
  • Manual test: airplane mode check-in → reconnect → confirm row appears in DB
Phase 4 — Ship
  • Manual smoke test of all 3 roles end-to-end
  • Vercel production deploy
  • Real event created, real officer list seeded (no demo data left behind)

05Anti Over-Engineering Reminders

Before adding anything

Ask: does this help answer "was this officer present, yes or no, recorded by whom"? If not, it's v2.

  • No CI/CD pipeline. You're solo — deploy from your machine via Vercel CLI or git push, review it yourself.
  • No automated test suite for v1. Manual smoke test per phase is enough at this scale.
  • No edit/delete on attendance rows. Append-only, corrections are new rows.
  • No analytics, streaks, leaderboards, or gamification — these always sound small and never are.
  • No multi-council/multi-org abstraction. Build for one council. Generalize later if it's ever actually needed.
  • No native app. PWA covers "add to home screen" and offline — that's the whole ask.
  • Resist RLS + Prisma both. Pick one enforcement layer (Prisma + API route checks) and stop there.
  • If an execution agent proposes a "flexible" or "extensible" abstraction you didn't ask for — reject it. Flexibility you don't need yet is a bug waiting to happen.

06Manual Setup Steps (you, not an agent)

1

Gather USC branding assets

Get the official USC logo (transparent PNG, at least 512×512) from your council's official Facebook page, past tarpaulins, or ask the outgoing secretary/PIO for the original file.

You'll need: 512×512 and 192×192 PNGs for the PWA manifest icons, plus a favicon.ico. If USC has official school colors, note the hex codes — use them for the app's theme_color instead of a generic palette.
2

Create the Supabase project

Sign up/log in at supabase.com, create a new project, choose a region close to the Philippines (Singapore is usually closest).

Save: project URL, anon key, service role key, and the session-pooler connection string (port 5432) for Prisma's DATABASE_URL.
3

Set up the Vercel project

Import the repo into Vercel, set framework preset to Vite, add environment variables (DATABASE_URL, SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY).

Mark service role key as a server-only env var — never expose it to the client bundle.
4

Get real event coordinates

For each recurring venue (session hall, gym, quadrangle), grab the lat/lng from Google Maps (right-click → coordinates) ahead of time so staff aren't guessing radius values on event-creation day.

Suggested starting radius: 50–75 meters for open areas like a quadrangle, 25–30 meters for smaller rooms.
5

Seed real officer accounts

Compile the actual officer roster (name, position, role: staff or client) before go-live — this is manual data entry only you or the secretary should own, not an agent.

6

Test geolocation permissions on real devices

Browser geolocation prompts differ across Android Chrome, iOS Safari, and in-app browsers (e.g. Facebook's in-app browser often blocks geolocation). Test check-in on at least one Android and one iPhone before the first real event.

7

Domain (optional)

If USC has or can get a subdomain (e.g. attendance.usc-yourschool.edu.ph or similar), point it at the Vercel deployment for credibility instead of a random .vercel.app URL.
